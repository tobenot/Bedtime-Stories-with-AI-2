# 机制包制作指南

> 文档版本：`202605150127`（按当前时间生成）

本文面向想自己做 GameMode 机制包的开发者，目标是：**用一份 JSON，定义状态、工具、触发器和开场体验**，并让运行时可直接加载。

## 1. 先理解机制包的边界

机制包只做"声明"，不写业务代码；执行由 GameMode Runtime 负责。

- **你负责**：`initialState`、`ui`、`pools`、`tools`、`triggers`、`instructions`。
- **运行时负责**：工具执行、触发器调度、状态补丁应用、结果回传 AI。

当前运行时支持的工具类型：

- `dice`
- `table`
- `encounter`
- `stateCheck`
- `patchState`

## 2. 机制包最小骨架（先跑起来）

先用这个最小模板，确认能导入、能开局、能看到状态面板：

```json
{
  "id": "my_first_pack",
  "title": "我的第一个机制包",
  "version": 202605150127,
  "description": "示例：带检定和随机遭遇的机制包",
  "turnPath": "world.turn",
  "instructions": {
    "narrator": "你是主持人，负责描述场景与 NPC 行为。",
    "rules": "当行动存在不确定性时请求工具，不要编造工具结果。"
  },
  "initialState": {
    "player": {
      "hp": 10,
      "maxHp": 10,
      "inventory": []
    },
    "world": {
      "turn": 0,
      "location": "起点",
      "locationTag": "town"
    }
  },
  "ui": [
    { "type": "stat", "label": "生命值", "path": "player.hp", "maxPath": "player.maxHp" },
    { "type": "text", "label": "地点", "path": "world.location" },
    { "type": "list", "label": "背包", "path": "player.inventory" }
  ],
  "pools": {},
  "tools": [],
  "triggers": [],
  "openingMessage": "你站在村口，夜色刚刚降临。"
}
```

## 3. 按这 6 步制作一个可玩机制包

### 第 1 步：定义状态模型（`initialState`）

建议先明确三类状态：

- **玩家状态**：如 `hp`、`gold`、`inventory`
- **世界状态**：如 `turn`、`location`、`locationTag`
- **剧情标记**：如 `world.flags.xxx`

建议保证 `turnPath` 指向一个数字字段（默认 `world.turn`）。

### 第 2 步：定义展示字段（`ui`）

`ui` 决定状态面板显示哪些内容，常用类型：

- `stat`：数值条（需 `path` + `maxPath`）
- `text`：文本
- `list`：列表

路径统一使用点号格式，例如：`player.hp`、`world.location`。

### 第 3 步：定义随机池（`pools`）

把会复用的候选项放进池里，推荐按领域命名：

- `encounter.actor`
- `encounter.intent`
- `encounter.complication`

每个词条可包含：

- `weight`：权重
- `text`：展示文本
- `tags`：标签
- `patch`：命中后状态补丁
- `conditions`：可用条件

### 第 4 步：定义工具（`tools`）

工具是"可执行能力"。建议先做两个：

1. 一个检定工具（`dice`）
2. 一个遭遇工具（`encounter`）

典型示例：

```json
{
  "tools": [
    {
      "id": "rollD20",
      "type": "dice",
      "label": "D20 检定",
      "visibility": ["ai", "manual"],
      "config": { "notation": "1d20" }
    },
    {
      "id": "randomEncounter",
      "type": "encounter",
      "label": "随机遭遇",
      "visibility": ["ai", "manual", "trigger"],
      "config": {
        "template": "{{actor.text}}正在{{intent.text}}。{{complication.text|}}",
        "rolls": {
          "actor": { "pool": "encounter.actor" },
          "intent": { "pool": "encounter.intent", "matchTagsFrom": ["actor", "actorTraits"] },
          "complication": { "pool": "encounter.complication", "chance": 0.5 }
        }
      }
    }
  ]
}
```

`visibility` 建议按用途设置：

- `ai`：允许 AI 请求
- `manual`：允许手动点击
- `trigger`：允许触发器调用

### 第 5 步：定义自动触发（`triggers`）

触发器用于"到条件就自动执行工具"。

```json
{
  "triggers": [
    {
      "id": "wild-every-3-turns",
      "label": "荒野随机遭遇",
      "toolId": "randomEncounter",
      "conditions": [
        { "path": "world.locationTag", "op": "eq", "value": "wild" },
        { "path": "world.turn", "op": "every", "value": 3 }
      ]
    }
  ]
}
```

常见 `op`：`eq`、`neq`、`gt`、`gte`、`lt`、`lte`、`includes`、`every`。

### 第 6 步：写主持人指令与开场（`instructions` + `openingMessage` + `loadingMessages`）

- `instructions.narrator`：身份、语气、叙述风格
- `instructions.rules`：裁定原则、何时请求工具、状态写回约束
- `openingMessage`：开局第一段文本
- `loadingMessages`：AI 思考期间显示的主题化提示（字符串数组，可选）

建议把"不得编造工具结果"写进 `rules`，确保工具闭环稳定。

`loadingMessages` 示例：

```json
{
  "loadingMessages": [
    "主持人正在翻阅怪物图鉴……",
    "前方的迷雾逐渐散去……",
    "骰子在桌面上旋转不止……",
    "酒馆的吟游诗人正在低声吟唱……"
  ]
}
```

建议 8-15 条，风格与机制包氛围一致，每条不超过 20 字。未声明时只显示默认的三点等待动画。

## 4. 可直接复制的进阶片段

### 4.1 Actor 私有特征池（`traits`）

当某些特征只属于特定单位时，放在 `actor.traits` 下，不污染全局池。

```json
{
  "id": "slime",
  "text": "史莱姆",
  "tags": ["beast", "hostile"],
  "traits": {
    "surface": [
      { "weight": 3, "text": "浑身覆盖黏液", "tags": ["sticky"] },
      { "weight": 1, "text": "散发酸性蒸汽", "tags": ["acid"], "patch": { "world.alertLevel": 1 } }
    ]
  }
}
```

模板可这样引用：`{{actorTraits.surface.text|无特殊状态}}`。

### 4.2 向后兼容旧字段（`prompts`）

当前加载器会把旧字段自动兼容到新结构：

- `prompts.host` / `prompts.narrator` → `instructions.narrator`
- `prompts.rules` → `instructions.rules`

新机制包建议直接使用 `instructions`。

## 5. 制作完成后的检查清单

发布前逐条检查：

- `id` 全局唯一且稳定
- `version` 为数字（推荐时间戳，例如 `202605150127`）
- `turnPath` 指向数字字段
- `ui.path` / `ui.maxPath` 均能在状态中解析
- `tools[*].id` 唯一，`type` 在支持列表内
- `triggers[*].toolId` 能在 `tools` 中找到
- `patch` 全部使用点号路径
- `openingMessage` 非空
- `loadingMessages` 若声明则为字符串数组，每条不超过 20 字（可选）

## 6. 推荐迭代顺序

- **V1**：仅 `dice` + 基础 `encounter`
- **V2**：加入 `triggers`（周期事件、区域事件）
- **V3**：加入 `traits`、标签联动、条件过滤
- **V4**：补充 `stateCheck` / `patchState` 做规则守门

---

如果你准备做"DND/COC 之外"的新题材（校园、赛博、武侠、末日），建议先复用本指南的最小骨架，再只替换 `initialState`、`pools`、`instructions`，这样上线速度最快、调试成本最低。
