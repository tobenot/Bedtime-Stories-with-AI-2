# 机制包制作指南

> 文档版本：`202605150127`（按当前时间生成）

本文面向想自己做 GameMode 机制包的开发者，目标是：**用一份 JSON，定义状态、工具、触发器和开场体验**，并让运行时可直接加载。

## 1. 先理解机制包的边界

机制包只做"声明"，不写业务代码；执行由 GameMode Runtime 负责。

- **你负责**：`initialState`、`ui`、`pools`、`tools`、`triggers`、`instructions`。
- **运行时负责**：工具执行、触发器调度、状态补丁应用、结果回传 AI。

当前运行时支持的工具类型：

- `dice`
- `table`
- `encounter`
- `stateCheck`
- `patchState`

## 2. 机制包最小骨架（先跑起来）

先用这个最小模板，确认能导入、能开局、能看到状态面板：

```json
{
  "id": "my_first_pack",
  "title": "我的第一个机制包",
  "version": 202605150127,
  "description": "示例：带检定和随机遭遇的机制包",
  "turnPath": "world.turn",
  "instructions": {
    "narrator": "你是主持人，负责描述场景与 NPC 行为。",
    "style": "简洁明快的叙述风格。每段 2-4 句，以动作和对话推进，少用长篇描写。",
    "rules": "当行动存在不确定性时请求工具，不要编造工具结果。"
  },
  "initialState": {
    "player": {
      "hp": 10,
      "maxHp": 10,
      "inventory": []
    },
    "world": {
      "turn": 0,
      "location": "起点",
      "locationTag": "town"
    }
  },
  "ui": [
    { "type": "stat", "label": "生命值", "path": "player.hp", "maxPath": "player.maxHp" },
    { "type": "text", "label": "地点", "path": "world.location" },
    { "type": "list", "label": "背包", "path": "player.inventory" }
  ],
  "pools": {},
  "tools": [],
  "triggers": [],
  "openingMessage": "你站在村口，夜色刚刚降临。"
}
```

## 3. 按这 6 步制作一个可玩机制包

### 第 1 步：定义状态模型（`initialState`）

建议先明确三类状态：

- **玩家状态**：如 `hp`、`gold`、`inventory`
- **世界状态**：如 `turn`、`location`、`locationTag`
- **剧情标记**：如 `world.flags.xxx`

建议保证 `turnPath` 指向一个数字字段（默认 `world.turn`）。

### 第 2 步：定义展示字段（`ui`）

`ui` 决定状态面板显示哪些内容，常用类型：

- `stat`：数值条（需 `path` + `maxPath`）
- `text`：文本
- `list`：列表

路径统一使用点号格式，例如：`player.hp`、`world.location`。

可见性相关字段：

- `ui[*].hidden`（可选，默认 `false`）：设为 `true` 后，该字段不会出现在玩家状态面板（含移动端回退摘要）。
- `hiddenStatePaths`（可选，`string[]`）：这些路径的状态变化不会写入玩家可见回复中的 `【状态变化】` 区块，但仍会保留在运行日志里。

推荐用法：如果你有“主持人内部状态”（如刺客动向、暗线计时器），把路径放进 `hiddenStatePaths`，并在 `ui` 里对应字段加上 `hidden: true`。


### 第 3 步：定义随机池（`pools`）

把会复用的候选项放进池里，推荐按领域命名：

- `encounter.actor`
- `encounter.intent`
- `encounter.complication`

每个词条可包含：

- `weight`：权重
- `text`：展示文本
- `tags`：标签
- `patch`：命中后状态补丁
- `conditions`：可用条件

### 第 4 步：定义工具（`tools`）

工具是"可执行能力"。建议先做两个：

1. 一个检定工具（`dice`）
2. 一个遭遇工具（`encounter`）

典型示例：

```json
{
  "tools": [
    {
      "id": "rollD20",
      "type": "dice",
      "label": "D20 检定",
      "visibility": ["ai", "manual"],
      "config": { "notation": "1d20" }
    },
    {
      "id": "randomEncounter",
      "type": "encounter",
      "label": "随机遭遇",
      "visibility": ["ai", "manual", "trigger"],
      "config": {
        "template": "{{actor.text}}正在{{intent.text}}。{{complication.text|}}",
        "rolls": {
          "actor": { "pool": "encounter.actor" },
          "intent": { "pool": "encounter.intent", "matchTagsFrom": ["actor", "actorTraits"] },
          "complication": { "pool": "encounter.complication", "chance": 0.5 }
        }
      }
    }
  ]
}
```

`visibility` 建议按用途设置：

- `ai`：允许 AI 请求
- `manual`：允许手动点击
- `trigger`：允许触发器调用

### 第 5 步：定义自动触发（`triggers`）

触发器用于"到条件就自动执行工具"。

```json
{
  "triggers": [
    {
      "id": "wild-every-3-turns",
      "label": "荒野随机遭遇",
      "toolId": "randomEncounter",
      "conditions": [
        { "path": "world.locationTag", "op": "eq", "value": "wild" },
        { "path": "world.turn", "op": "every", "value": 3 }
      ]
    }
  ]
}
```

常见 `op`：`eq`、`neq`、`gt`、`gte`、`lt`、`lte`、`includes`、`every`。

### 第 6 步：写主持人指令、文风与开场（`instructions` + `openingMessage`）

`instructions` 有三个维度，各管一件事：

| 字段 | 管什么 | 示例问题 |
|---|---|---|
| `narrator` | **你是谁** — 身份、人称、语气 | "你是哥特风主持人" / "你是一位冷静的调查局档案员" |
| `style` | **你怎么写** — 叙事风格、用词偏好、节奏、篇幅 | "短句、意象密集" / "书面白话，适度文言点缀" |
| `rules` | **你守什么规矩** — 裁定原则、何时请求工具、状态约束 | "不得编造工具结果" / "检定 ≥ 难度值为成功" |

> **为什么把 narrator 和 style 分开？**
>
> `narrator` 定义角色身份（"你是一位阴郁的主持人"），`style` 定义文学表达方式（"用短句、多感官描写"）。同一个身份可以搭配不同文风（比如"哥特主持人 + 简洁风"和"哥特主持人 + 华丽风"），分开声明让调试和复用更方便。

#### 文风设计建议

文风（`instructions.style`）是影响玩家阅读体验最直接的字段。以下是设计要点：

**需要明确的维度：**

- **句式**：长句还是短句？简单句为主还是复合句？
- **用词偏好**：口语化 / 书面化 / 文言混搭？意象密集还是直白？
- **感官偏重**：偏视觉描写？听觉？触感？气味？
- **节奏感**：快节奏推进 / 慢节奏沉浸 / 张弛交替？
- **篇幅控制**：每段叙述大约几句话？留白多还是铺陈多？
- **修辞偏好**：是否大量使用比喻、拟人、通感？

**不同题材的文风参考：**

```json
{
  "style": "用短句、意象密集的方式叙述。多用感官描写（声音、气味、触感），少用抽象概括。每段叙述控制在 3-5 句以内，留白多于铺陈。战斗场景节奏加快，探索场景节奏放慢。"
}
```

```json
{
  "style": "使用悬疑小说的笔法。叙述时刻意隐藏关键信息，只给出感官层面的线索。多用省略号和短句制造不安感。每段 2-4 句，绝不超过 5 句。避免直接揭示恐怖，用暗示替代。"
}
```

```json
{
  "style": "轻松的校园日常文风。对话为主，叙述为辅。用口语化的短句，偶尔插入吐槽和网络用语。情绪表达直接，不用隐喻。每段 2-3 句即可。"
}
```

```json
{
  "style": "书面白话为主，适度点缀文言句式。描写场景时注重声色气味的层次感。战斗描写借鉴武侠小说的招式感，快节奏短句连击。日常场景用散文笔法，不急不缓。"
}
```

> 如果机制包没有声明 `style`，运行时不会注入 `[style]` 段。AI 会使用默认的"通用叙述"风格——能用，但没有辨识度。**强烈建议每个机制包都写 style**。

#### 完整示例

```json
{
  "instructions": {
    "narrator": "你是一位阴郁而诗意的哥特风主持人。第二人称叙述。",
    "style": "用短句、意象密集的方式叙述。多用感官描写（声音、气味、触感），少用抽象概括。每段叙述控制在 3-5 句以内，留白多于铺陈。",
    "rules": "当行动存在不确定性时请求工具，不要编造工具结果。玩家每次行动消耗 1 点行动力。"
  },
  "openingMessage": "蜡烛的火焰在门缝的穿堂风中摇曳。你站在塔楼的入口，空气里弥漫着陈年霉菌和铁锈的气味。楼梯向上延伸，消失在视线之外的黑暗中。"
}
```

建议把"不得编造工具结果"写进 `rules`，确保工具闭环稳定。

## 4. 可直接复制的进阶片段

### 4.1 Actor 私有特征池（`traits`）

当某些特征只属于特定单位时，放在 `actor.traits` 下，不污染全局池。

```json
{
  "id": "slime",
  "text": "史莱姆",
  "tags": ["beast", "hostile"],
  "traits": {
    "surface": [
      { "weight": 3, "text": "浑身覆盖黏液", "tags": ["sticky"] },
      { "weight": 1, "text": "散发酸性蒸汽", "tags": ["acid"], "patch": { "world.alertLevel": 1 } }
    ]
  }
}
```

模板可这样引用：`{{actorTraits.surface.text|无特殊状态}}`。

### 4.2 向后兼容旧字段（`prompts`）

当前加载器会把旧字段自动兼容到新结构：

- `prompts.host` / `prompts.narrator` → `instructions.narrator`
- `prompts.rules` → `instructions.rules`

新机制包建议直接使用 `instructions`。

## 5. 制作完成后的检查清单

发布前逐条检查：

- `id` 全局唯一且稳定
- `version` 为数字时间戳（推荐 `YYYYMMDDHHmm`，例如 `202605150212`；同一分钟多次改动可用 `YYYYMMDDHHmmss`）
- `turnPath` 指向数字字段
- `ui.path` / `ui.maxPath` 均能在状态中解析
- `tools[*].id` 唯一，`type` 在支持列表内
- `triggers[*].toolId` 能在 `tools` 中找到
- `patch` 全部使用点号路径
- `openingMessage` 非空
- `instructions.style` 已声明且具体（不要只写"生动有趣"，要写清句式、篇幅、感官偏好等具体维度）
- `openingMessage` 的文风与 `instructions.style` 一致（开场是玩家看到的第一段文字，风格应与后续 AI 输出统一）

## 6. 推荐迭代顺序

- **V1**：仅 `dice` + 基础 `encounter`
- **V2**：加入 `triggers`（周期事件、区域事件）
- **V3**：加入 `traits`、标签联动、条件过滤
- **V4**：补充 `stateCheck` / `patchState` 做规则守门

---

如果你准备做"DND/COC 之外"的新题材（校园、赛博、武侠、末日），建议先复用本指南的最小骨架，再只替换 `initialState`、`pools`、`instructions`，这样上线速度最快、调试成本最低。

## 附录：增量更新（`202605150210`）

本次新增机制包可选字段 `loadingMessages`，用于 AI 思考期间展示主题化等待提示。

- 字段类型：`string[]`（可选）
- 建议数量：8-15 条
- 文案建议：每条不超过 20 字，风格与机制包题材一致
- 运行时行为：AI 响应等待时随机显示，并每 4 秒轮换；未声明则仅显示默认三点动画

示例：

```json
{
  "loadingMessages": [
    "主持人正在翻阅怪物图鉴……",
    "前方的迷雾逐渐散去……",
    "骰子在桌面上旋转不止……",
    "酒馆的吟游诗人正在低声吟唱……"
  ]
}
```
