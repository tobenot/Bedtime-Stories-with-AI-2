# 存档与导入

GameMode 需要同时保存玩家可读文本和结构化运行状态。

## 对话文本

对话文本应保证普通导出也可阅读。

示例：

```text
你压低身形靠近营地，但脚下枯枝发出清脆的断裂声。

【检定】
D20 检定：1d20 = 7，难度 12，失败。

【状态变化】
world.alertLevel：0 → 1

【可选行动】
1. 立刻躲进旁边的灌木
2. 装作路过的旅人
3. 抢先制服守卫
```

## 结构化状态

结构化状态保存在：

```js
chat.metadata.gameMode = {
	packId: 'builtin_dnd_adventure',
	state: {},
	triggerState: {}
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `packId` | 当前机制包 ID |
| `state` | 当前游戏状态 |
| `triggerState` | 触发器运行状态，例如触发次数、上次触发回合 |

## 消息事件元数据

每条 GameMode assistant 消息建议保存：

```js
message.metadata.gameEvent = {
	phase: 'final',
	toolRequests: [],
	toolResults: [],
	triggerResults: [],
	stateChanges: [],
	choices: [],
	stateSnapshot: {}
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `phase` | 最终阶段，通常为 `final` |
| `toolRequests` | AI 请求过的工具 |
| `toolResults` | 工具执行结果 |
| `triggerResults` | 本回合触发器结果 |
| `stateChanges` | 状态变化记录 |
| `choices` | 本轮可选行动 |
| `stateSnapshot` | **本回合结束后的完整游戏状态快照** |

## 状态快照

### 设计原则

**状态快照跟随消息存档**。每条游戏回合消息保存该回合结束时的完整状态副本。玩家随时可以从任意历史消息分支出新对话，新对话的游戏状态从该消息的快照恢复，而非从 initialState 重建。

### 为什么快照绑定消息

1. **分支天然正确**：从中间消息 fork 时，直接从该消息的 `stateSnapshot` 恢复，不会状态错位。
2. **Undo/Redo 与回放统一**：撤销、重做、历史回看都基于消息快照，只有一个状态真相源。
3. **调试友好**：任何异常都能定位到"哪条消息产生了哪次状态变化"。
4. **无需重放**：不依赖从 `initialState` 逐条重放 `stateChanges`，避免重放链断裂风险。

### 快照存储策略

采用**逐条全量快照**策略，即每条 assistant 消息的 `gameEvent.stateSnapshot` 保存该回合结束时的完整 `gameState` + `triggerState` 深拷贝：

```js
message.metadata.gameEvent.stateSnapshot = {
	state: { /* 完整 gameState 深拷贝 */ },
	triggerState: { /* 完整 triggerState 深拷贝 */ }
}
```

**为什么不用增量 patch**：

- 典型机制包状态总量 < 2KB，50 回合全量快照约 100KB，存储开销可接受。
- 全量快照无需重放链，任意节点可独立恢复，实现简单、可靠性高。
- 避免了增量 patch 链断裂时无法恢复的灾难性故障。

### 快照保存时机

| 消息类型 | 是否保存快照 | 说明 |
|---|---|---|
| `gameResponse`（phase=final） | ✅ 保存 | 正常游戏回合，状态已更新 |
| `plainResponse`（phase=plain） | ✅ 保存 | AI 未返回 JSON 但触发器可能已改状态 |
| `manualTool`（手动工具） | ✅ 保存 | 手动工具可能改变状态 |
| `opening`（开场消息） | ✅ 保存 | 记录 initialState 基线 |
| user 消息 | ❌ 不保存 | 用户消息不改变游戏状态 |

### 从快照恢复状态

**分支（Fork）**：

```js
// 从目标消息的快照恢复
const snapshot = targetMessage.metadata?.gameEvent?.stateSnapshot;
if (snapshot) {
	newChat.metadata.gameMode.state = cloneData(snapshot.state);
	newChat.metadata.gameMode.triggerState = cloneData(snapshot.triggerState);
}
```

**回退（Undo）**：

```js
// 找到上一条有快照的 assistant 消息
const prevSnapshot = findLastSnapshot(messages, currentIndex - 1);
if (prevSnapshot) {
	gameMode.state = cloneData(prevSnapshot.state);
	gameMode.triggerState = cloneData(prevSnapshot.triggerState);
}
```

### 向后兼容

旧消息没有 `stateSnapshot` 字段时：

- Fork / Undo 尝试从 `stateChanges` 逆向重建（best effort），如果 `stateChanges` 不完整则回退到 `initialState`。
- 不需要迁移旧数据，新消息自动带快照。

## 工具结果保存

工具结果不能只保存为文本。应同时保存结构化数据：

```json
{
	"toolId": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"reason": "潜行检定",
	"ok": true,
	"displayText": "D20 检定：1d20 = 7",
	"data": {
		"notation": "1d20",
		"rolls": [7],
		"total": 7,
		"difficulty": 12,
		"outcome": "failure"
	},
	"changes": []
}
```

这样后续可以做：

- 回放游戏过程
- 调试 AI 裁定
- 恢复状态
- 统计骰子和随机表结果
- 生成战报或日志

## 导入机制包

游戏模式左侧可以导入 `.json`。

支持：

- 单个机制包对象
- 机制包数组

导入后写入浏览器本地：

```text
bs2_custom_game_packs
```

## 兼容旧存档

如果旧存档只有：

```js
chat.metadata.gameMode = {
	packId: 'builtin_dnd_adventure',
	state: {}
}
```

加载时应补齐：

```js
chat.metadata.gameMode.triggerState ||= {}
```

如果旧消息没有 `metadata.gameEvent`，不需要迁移。它们仍然作为普通历史文本参与上下文。
