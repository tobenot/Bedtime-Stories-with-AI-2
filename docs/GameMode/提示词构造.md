# 提示词构造

提示词管理器负责将机制包声明、对话历史、当前状态和工具结果拼装为最终发送给 AI 的 messages 数组。

核心设计目标：**最大化前缀缓存命中率**。

## 为什么不用 system prompt

项目使用 OpenAI 兼容接口（原生 fetch），不使用 function calling，也不依赖 API 级别的 `tools` 或单独的 `system` 字段。所有内容都通过 `messages` 数组传递。

大模型的前缀缓存（Prompt Cache / KV Cache）按 messages 数组从前到后做精确前缀匹配：

- 前缀部分完全相同 → 命中缓存，跳过重新计算
- 前缀中任何位置发生变化 → 从变化点开始全部重算

因此：

> **不变的内容放前面，变化的内容放后面。整个 messages 数组就是缓存单元。**

如果把每轮都变的状态、触发器结果放进 `role: "system"` 并置于数组首位，等于每轮都把前缀打碎，缓存命中率为零。

## 分层模型

将所有内容按**变化频率**分为三层：

```text
┌─────────────────────────────────────────────────┐
│  前缀层（Prefix）— 整局不变                       │
│  对话的第一条 user message 包含：                  │
│    ├─ instructions.narrator  主持人身份/风格       │
│    ├─ instructions.rules     规则/裁定原则         │
│    ├─ instructions.format    输出格式约束          │
│    └─ instructions.toolDefs  完整工具定义列表      │
├─────────────────────────────────────────────────┤
│  对话层（History）— 只追加不修改                   │
│    ├─ assistant msg 1                            │
│    ├─ user msg 2                                 │
│    ├─ assistant msg 2                            │
│    └─ ...                                        │
├─────────────────────────────────────────────────┤
│  后缀层（Suffix）— 每轮都变                       │
│  对话的最后一条 user message 包含：                │
│    ├─ 当前状态快照 state                          │
│    ├─ 本轮可用工具子集 availableTools（如需动态过滤）│
│    ├─ 本轮触发器执行结果 triggerResults            │
│    ├─ 本轮工具执行结果 toolResults                 │
│    ├─ 控制指令（如 forceFinal）                   │
│    └─ 玩家本轮输入 playerInput                    │
└─────────────────────────────────────────────────┘
```

实际发送给 API 的 messages 结构：

```json
[
  { "role": "user", "content": "<前缀层：instructions 全文>" },
  { "role": "assistant", "content": "<开场回复>" },
  { "role": "user", "content": "玩家第1轮输入" },
  { "role": "assistant", "content": "AI第1轮回复" },
  { "role": "user", "content": "玩家第2轮输入" },
  { "role": "assistant", "content": "AI第2轮回复" },
  { "role": "user", "content": "<后缀层：状态+触发器+工具结果+本轮玩家输入>" }
]
```

前缀层 + 对话层构成稳定前缀，只有最后一条 user message 每轮变化。在 20 轮游戏中，90%+ 的 token 可以命中缓存。

## 字段命名

旧字段 `prompts.host` / `prompts.rules` 容易与 API 的 system prompt 概念混淆。重新命名为 `instructions`，语义为"发给 AI 的指令"，不绑定 API 任何特定字段：

| 旧字段 | 新字段 | 说明 |
|---|---|---|
| `prompts.host` | `instructions.narrator` | 主持人身份、语气、风格 |
| `prompts.rules` | `instructions.rules` | 游戏规则、裁定原则、世界机制 |
| （隐式写死） | `instructions.format` | 输出格式约束（JSON schema） |
| （隐式写死） | `instructions.toolDefs` | 工具定义列表（AI 可见的完整描述） |

### 向后兼容

加载时做归一化：

```js
// 兼容旧字段
if (pack.prompts && !pack.instructions) {
  pack.instructions = {
    narrator: pack.prompts.host || pack.prompts.narrator || '',
    rules: pack.prompts.rules || ''
  };
}
```

### 机制包示例

```json
{
  "id": "my_pack",
  "title": "我的机制包",
  "instructions": {
    "narrator": "你是一位阴郁而诗意的哥特风主持人。用短句、意象密集的方式叙述。",
    "rules": "玩家每次行动消耗 1 点行动力。行动力归零时本日结束。检定使用 D20，≥难度值为成功。"
  }
}
```

## 前缀层构造细节

前缀层作为第一条 user message 发送，内容在开局时确定，整局不变：

```text
[narrator]
{instructions.narrator}

[rules]
{instructions.rules}

[format]
你是文本游戏主持人。根据玩家输入推进游戏。
只能返回 JSON，不要包裹 Markdown 代码块。
如果需要检定、随机表或其他工具，先返回 phase=tool_request，不要编造工具结果。
当已有工具结果足以裁定，或不需要工具时，返回 phase=final。
返回格式：{"phase":"...","toolRequests":[...],"narration":"...","choices":[...],"statePatch":{}}

[tools]
可用工具定义：
- rollD20 (dice)：D20 检定，输入 {reason, difficulty?, modifier?}
- randomEncounter (encounter)：随机遭遇
...
```

注意：这里放的是工具的**完整定义**（因为不变），而不是"当前可用工具子集"。如果某些工具需要根据状态动态解锁，在后缀层用一句话声明即可：

```text
本轮可用工具：[rollD20, randomEncounter]
```

这样前缀不被破坏。

## 后缀层构造细节

后缀层是每轮最后一条 user message，包含所有动态信息：

```text
[state]
当前状态：{"player":{"hp":7,"maxHp":10,"inventory":["火把"]},"world":{"turn":5,"location":"废弃塔楼","locationTag":"wild"}}

[triggers]
本回合触发器结果：
【荒野随机遭遇】随机遭遇：actor=饥饿的狼；intent=正在追踪血迹

[toolResults]
本回合已执行工具结果：
【D20 检定】1d20 = 7，难度 12，失败

[directive]
本回合工具调用次数已达上限。必须返回 phase=final。

[playerInput]
玩家行动：我悄悄潜入营地。
```

各段按需拼接，空段跳过。

## Tool Loop 二次调用时的缓存

当 AI 返回 `phase=tool_request` 后需要二次调用，此时：

1. 前缀层不变 ✅
2. 对话层不变（因为本轮还没写入新的 assistant message）✅
3. 后缀层追加新的 toolResults 后重发

所以二次调用时，前缀仍然命中。只有后缀层多了一段工具结果文本。

## 与现有代码的对应

| 当前代码 | 改造方向 |
|---|---|
| `buildGameSystemPrompt()` 生成一整个 system content | 拆分为 `buildPrefixMessage()` 和 `buildSuffixMessage()` |
| system 放在 messages[0] | prefix 作为第一条 user message，开局时生成一次 |
| 状态、触发器结果塞在 system 里 | 移到最后一条 user message（suffix） |
| `callGameAi()` 每次重建整个数组 | prefix 缓存在内存中，只重建 suffix |

## 缓存收益估算

假设一局游戏 20 轮，每轮 prefix 约 800 token，history 平均 200 token/轮：

| 轮次 | 总 token | 可缓存 prefix | 缓存命中率 |
|---|---|---|---|
| 第 1 轮 | ~1000 | 0（首次写入） | 0% |
| 第 5 轮 | ~1800 | ~1600 | ~89% |
| 第 10 轮 | ~2800 | ~2600 | ~93% |
| 第 20 轮 | ~4800 | ~4600 | ~96% |

对于支持缓存计费的模型（如 Claude），缓存部分费用为原价 10%，综合可节省 80%+ 的输入 token 费用。

## 设计决策记录

1. **为什么不用 role: "system"？** — 我们用的是 OpenAI 兼容接口，system 只是 messages 数组的第一项。如果 system 内容每轮变化，前缀全部失效。不如把静态内容放在第一条 user message 中实现同样效果，且对缓存友好。

2. **为什么字段叫 instructions 不叫 prompts？** — `prompts` 容易和 API 概念混淆（system prompt）。`instructions` 更准确表达"给 AI 的指令"，且不暗示必须放在 system 位置。

3. **为什么工具定义放前缀而不放后缀？** — 工具定义在整局中不变。如果需要动态控制可用性，在后缀层注入一行"本轮可用工具列表"即可。前缀保持稳定，缓存不会被破坏。

4. **玩家输入为什么放后缀最后？** — 确保所有工具结果和状态在玩家输入之前注入，AI 先看到完整上下文再看玩家动作。
