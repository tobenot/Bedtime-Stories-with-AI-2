# 提示词构造

提示词管理器负责将机制包声明、对话历史、当前状态和工具结果拼装为最终发送给 AI 的 messages 数组。

核心设计目标：**最大化前缀缓存命中率**。

## 为什么不用 system prompt

项目使用 OpenAI 兼容接口（原生 fetch），不使用 function calling，也不依赖 API 级别的 `tools` 或单独的 `system` 字段。所有内容都通过 `messages` 数组传递。

大模型的前缀缓存（Prompt Cache / KV Cache）按 messages 数组从前到后做精确前缀匹配：

- 前缀部分完全相同 → 命中缓存，跳过重新计算
- 前缀中任何位置发生变化 → 从变化点开始全部重算

因此：

> **不变的内容放前面，变化的内容放后面。整个 messages 数组就是缓存单元。**

如果把每轮都变的状态、触发器结果放进 `role: "system"` 并置于数组首位，等于每轮都把前缀打碎，缓存命中率为零。

## 分层模型

将所有内容按**变化频率**分为三层：

```text
┌─────────────────────────────────────────────────┐
│  前缀层（Prefix）— 整局不变                       │
│  对话的第一条 user message 包含：                  │
│    ├─ instructions.narrator   主持人身份/语气      │
│    ├─ instructions.style      文风（机制包各异）    │
│    ├─ continuity              叙事连贯性约束（共享）│
│    ├─ instructions.rules      规则/裁定原则        │
│    ├─ instructions.format     输出格式约束         │
│    └─ instructions.toolDefs   完整工具定义列表     │
├─────────────────────────────────────────────────┤
│  对话层（History）— 只追加不修改                   │
│    ├─ assistant msg 1                            │
│    ├─ user msg 2                                 │
│    ├─ assistant msg 2                            │
│    └─ ...                                        │
├─────────────────────────────────────────────────┤
│  后缀层（Suffix）— 每轮都变                       │
│  对话的最后一条 user message 包含：                │
│    ├─ 当前状态快照 state                          │
│    ├─ 本轮可用工具子集 availableTools（如需动态过滤）│
│    ├─ 本轮触发器执行结果 triggerResults            │
│    ├─ 本轮工具执行结果 toolResults                 │
│    ├─ 控制指令（如 forceFinal）                   │
│    └─ 玩家本轮输入 playerInput                    │
└─────────────────────────────────────────────────┘
```

实际发送给 API 的 messages 结构：

```json
[
  { "role": "user", "content": "<前缀层：instructions 全文>" },
  { "role": "assistant", "content": "<开场回复>" },
  { "role": "user", "content": "玩家第1轮输入" },
  { "role": "assistant", "content": "AI第1轮回复" },
  { "role": "user", "content": "玩家第2轮输入" },
  { "role": "assistant", "content": "AI第2轮回复" },
  { "role": "user", "content": "<后缀层：状态+触发器+工具结果+本轮玩家输入>" }
]
```

前缀层 + 对话层构成稳定前缀，只有最后一条 user message 每轮变化。在 20 轮游戏中，90%+ 的 token 可以命中缓存。

## 字段命名

旧字段 `prompts.host` / `prompts.rules` 容易与 API 的 system prompt 概念混淆。重新命名为 `instructions`，语义为"发给 AI 的指令"，不绑定 API 任何特定字段：

| 旧字段 | 新字段 | 说明 |
|---|---|---|
| `prompts.host` | `instructions.narrator` | 主持人身份、语气 |
| （新增） | `instructions.style` | 文风：叙事风格、用词偏好、节奏感（每个机制包各异） |
| `prompts.rules` | `instructions.rules` | 游戏规则、裁定原则、世界机制 |
| （隐式写死） | `instructions.format` | 输出格式约束（JSON schema） |
| （隐式写死） | `instructions.toolDefs` | 工具定义列表（AI 可见的完整描述） |
| （运行时注入） | continuity | 叙事连贯性约束（所有机制包共享，不在机制包中声明） |

### 向后兼容

加载时做归一化：

```js
// 兼容旧字段
if (pack.prompts && !pack.instructions) {
  pack.instructions = {
    narrator: pack.prompts.host || pack.prompts.narrator || '',
    rules: pack.prompts.rules || ''
  };
}
```

### 机制包示例

```json
{
  "id": "my_pack",
  "title": "我的机制包",
  "instructions": {
    "narrator": "你是一位阴郁而诗意的哥特风主持人。",
    "style": "用短句、意象密集的方式叙述。多用感官描写（声音、气味、触感），少用抽象概括。每段叙述控制在 3-5 句以内，留白多于铺陈。",
    "rules": "玩家每次行动消耗 1 点行动力。行动力归零时本日结束。检定使用 D20，≥难度值为成功。"
  }
}
```

## 叙事连贯性约束（基础要求）

这是所有机制包共享的底层叙事规则，注入前缀层，优先级高于单个机制包的文风设定。

### 问题背景

AI 每轮收到后缀层的**状态 JSON** 后，容易出现"看到状态就想写两句"的行为：看到 `location: "废弃塔楼"` 就自由发挥塔楼描写，而不是接着上一轮的故事线继续讲。这导致叙事上下文断裂，玩家体验不连贯。

### 核心规则

以下约束作为前缀层 `[continuity]` 段注入，位于 `[narrator]` 和 `[style]` 之后、`[format]` 之前：

```text
[continuity]
叙事连贯性要求（强制）：
1. 你的每段 narration 必须严格接续对话历史中最后一段叙述的剧情线。不要跳跃、不要重复已经描写过的场景，不要凭空引入未经铺垫的内容。
2. 状态 JSON 是机械记录，不是写作提纲。不要看到状态字段就展开描写——只在状态变化与当前剧情动作直接相关时才自然地融入叙述。
3. 工具结果和触发器结果是本轮的"事实素材"，必须基于这些素材推进叙事，不要忽略也不要另起炉灶。
4. 如果本轮没有显著剧情推进（例如玩家只是查看状态），用一两句过渡即可，不要强行铺陈。
```

### 设计意图

| 条目 | 解决的问题 |
|---|---|
| 第 1 条 | 防止 AI 每轮重新"开局"，确保故事线是一条线 |
| 第 2 条 | 防止 AI 把状态 JSON 当作写作灵感来源，看到什么写什么 |
| 第 3 条 | 确保工具/触发器结果被正确消化，而不是被叙事替代 |
| 第 4 条 | 防止"没事也要写一大段"的填充行为 |

### 与机制包文风的关系

连贯性约束是**结构性要求**（"怎么接"），文风是**风格性要求**（"怎么写"）。两者正交：

- 连贯性约束：所有机制包共享，写在运行时前缀层
- 文风（`instructions.style`）：每个机制包各自声明，也写在前缀层

即使机制包没有声明文风，连贯性约束仍然生效。

## 前缀层构造细节

前缀层作为第一条 user message 发送，内容在开局时确定，整局不变：

```text
[narrator]
{instructions.narrator}

[style]
{instructions.style}

[continuity]
叙事连贯性要求（强制）：
1. 你的每段 narration 必须严格接续对话历史中最后一段叙述的剧情线。不要跳跃、不要重复已经描写过的场景，不要凭空引入未经铺垫的内容。
2. 状态 JSON 是机械记录，不是写作提纲。不要看到状态字段就展开描写——只在状态变化与当前剧情动作直接相关时才自然地融入叙述。
3. 工具结果和触发器结果是本轮的"事实素材"，必须基于这些素材推进叙事，不要忽略也不要另起炉灶。
4. 如果本轮没有显著剧情推进（例如玩家只是查看状态），用一两句过渡即可，不要强行铺陈。

[rules]
{instructions.rules}

[format]
你是文本游戏主持人。根据玩家输入推进游戏。
只能返回 JSON，不要包裹 Markdown 代码块。
如果需要检定、随机表或其他工具，先返回 phase=tool_request，不要编造工具结果。
当已有工具结果足以裁定，或不需要工具时，返回 phase=final。
toolRequests.input 使用键值数组；无参数时返回空数组。statePatch 使用 { path, value } 数组；无状态变化时返回空数组。
返回格式：{"phase":"...","toolRequests":[{"toolId":"...","reason":"...","input":[{"key":"difficulty","value":12}]}],"narration":"...","choices":[...],"statePatch":[{"path":"world.location","value":"森林"}]}

[tools]
可用工具定义：
- rollD20 (dice)：D20 检定，输入 {reason, difficulty?, modifier?}
- randomEncounter (encounter)：随机遭遇
...
```

注意：这里放的是工具的**完整定义**（因为不变），而不是"当前可用工具子集"。如果某些工具需要根据状态动态解锁，在后缀层用一句话声明即可：

```text
本轮可用工具：[rollD20, randomEncounter]
```

这样前缀不被破坏。

## 后缀层构造细节

后缀层是每轮最后一条 user message，包含所有动态信息：

```text
[state]
当前状态：{"player":{"hp":7,"maxHp":10,"inventory":["火把"]},"world":{"turn":5,"location":"废弃塔楼","locationTag":"wild"}}

[triggers]
本回合触发器结果：
【荒野随机遭遇】随机遭遇：actor=饥饿的狼；intent=正在追踪血迹

[toolResults]
本回合已执行工具结果：
【D20 检定】1d20 = 7，难度 12，失败

[directive]
本回合工具调用次数已达上限。必须返回 phase=final。

[playerInput]
玩家行动：我悄悄潜入营地。
```

各段按需拼接，空段跳过。

## 结构化输出约束

OpenAI 兼容接口会优先通过请求体 `response_format: { type: "json_schema" }` 启用结构化输出。该 schema **不写入 messages**，因此不会污染长期对话上下文；它只作为本次 API 请求的输出约束发送。

当前 schema 只约束协议外壳：

- `phase`：`tool_request` 或 `final`
- `toolRequests`：工具请求数组，`input` 使用 `[{ "key": "参数名", "value": 参数值 }]`
- `narration`：玩家可见正文，`tool_request` 阶段为空字符串；**禁止**在该字段里写“可选行动/选项列表”
- `choices`：可选行动数组（仅用于前端独立 UI 的行动建议，不拼接进 `narration` 正文）

- `statePatch`：状态补丁数组，格式为 `[{ "path": "world.location", "value": "森林" }]`

`choices` 属于**临时建议数据**：每轮请求返回后仅在当前输入区展示，玩家可忽略并自由输入；玩家提交后即清空，不写入对话正文，也不参与后续上下文构造。


如果上游不支持 `json_schema`，运行时会先降级到 `json_object`；如果连 `response_format` 都不支持，再降级为普通 JSON 提示词，并继续使用容错解析与协议泄漏拦截。

## Tool Loop 二次调用时的缓存

当 AI 返回 `phase=tool_request` 后需要二次调用，此时：

1. 前缀层不变 ✅
2. 对话层不变（因为本轮还没写入新的 assistant message）✅
3. 后缀层追加新的 toolResults 后重发

所以二次调用时，前缀仍然命中。只有后缀层多了一段工具结果文本。

## 与现有代码的对应

| 当前代码 | 改造方向 |
|---|---|
| `buildGameSystemPrompt()` 生成一整个 system content | 拆分为 `buildPrefixMessage()` 和 `buildSuffixMessage()` |
| system 放在 messages[0] | prefix 作为第一条 user message，开局时生成一次 |
| 状态、触发器结果塞在 system 里 | 移到最后一条 user message（suffix） |
| `callGameAi()` 每次重建整个数组 | prefix 缓存在内存中，只重建 suffix |

## 缓存收益估算

假设一局游戏 20 轮，每轮 prefix 约 800 token，history 平均 200 token/轮：

| 轮次 | 总 token | 可缓存 prefix | 缓存命中率 |
|---|---|---|---|
| 第 1 轮 | ~1000 | 0（首次写入） | 0% |
| 第 5 轮 | ~1800 | ~1600 | ~89% |
| 第 10 轮 | ~2800 | ~2600 | ~93% |
| 第 20 轮 | ~4800 | ~4600 | ~96% |

对于支持缓存计费的模型（如 Claude），缓存部分费用为原价 10%，综合可节省 80%+ 的输入 token 费用。

## 设计决策记录

1. **为什么不用 role: "system"？** — 我们用的是 OpenAI 兼容接口，system 只是 messages 数组的第一项。如果 system 内容每轮变化，前缀全部失效。不如把静态内容放在第一条 user message 中实现同样效果，且对缓存友好。

2. **为什么字段叫 instructions 不叫 prompts？** — `prompts` 容易和 API 概念混淆（system prompt）。`instructions` 更准确表达"给 AI 的指令"，且不暗示必须放在 system 位置。

3. **为什么工具定义放前缀而不放后缀？** — 工具定义在整局中不变。如果需要动态控制可用性，在后缀层注入一行"本轮可用工具列表"即可。前缀保持稳定，缓存不会被破坏。

4. **玩家输入为什么放后缀最后？** — 确保所有工具结果和状态在玩家输入之前注入，AI 先看到完整上下文再看玩家动作。

5. **为什么需要 `[continuity]` 段？** — 实测发现 AI 收到后缀层的状态 JSON 后，容易"看到状态就想写两句"，导致叙事上下文断裂。连贯性约束明确告诉 AI：状态是机械记录不是写作提纲，narration 必须严格接续上一段剧情。这是所有机制包共享的基础要求，写死在运行时里。

6. **为什么把 narrator 和 style 分开？** — `narrator` 定义"你是谁"（身份、人称、语气），`style` 定义"你怎么写"（用词、节奏、修辞手法、篇幅控制）。分离后：(1) 同一个 narrator 身份可以搭配不同文风（比如同一个"哥特主持人"可以搭配"简洁"或"华丽"两种文风）；(2) 文风的独立字段让机制包作者更容易意识到这是一个需要认真设计的维度，而不是塞在 narrator 里被忽略。
