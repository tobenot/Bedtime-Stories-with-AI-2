# 工具基建设计

GameMode 工具是游戏模式内部能力，不设计为全局通用工具系统。

工具的目标是让机制包声明可执行规则，让 AI、用户和触发器都能在 GameMode 内部使用同一套运行期。

## 总体结构

```text
GameMode
  ├─ GamePack tools 声明
  ├─ Tool Registry 工具注册表
  ├─ Tool Runtime 工具执行器
  ├─ Tool Planner AI 工具规划
  ├─ Tool Loop 工具闭环
  └─ Tool Result Formatter 展示与存档格式化
```

## 工具生命周期

```text
机制包声明工具
  ↓
GameMode 加载机制包
  ↓
归一化工具定义
  ↓
根据 visibility 暴露给 AI / 手动按钮 / 触发器
  ↓
工具被请求
  ↓
Tool Runtime 校验、执行、应用 patch
  ↓
返回结构化 ToolResult
  ↓
Tool Loop 回传 AI 或写入消息
```

## 工具定义

推荐工具定义结构：

```json
{
	"id": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"description": "用于能力检定、攻击检定或豁免检定。",
	"visibility": ["ai", "manual", "trigger"],
	"inputSchema": {
		"reason": "string",
		"difficulty": "number?"
	},
	"config": {
		"notation": "1d20"
	}
}
```

兼容当前简单写法：

```json
{
	"id": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"notation": "1d20"
}
```

加载时归一化为：

```json
{
	"id": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"description": "",
	"visibility": ["ai", "manual", "trigger"],
	"inputSchema": {},
	"config": {
		"notation": "1d20"
	}
}
```

## `visibility`

`visibility` 决定工具可以被谁使用。

| 值 | 说明 |
|---|---|
| `ai` | AI 可以主动请求 |
| `manual` | 玩家可以在界面按钮中手动执行 |
| `trigger` | 触发器可以自动执行 |

示例：

```json
{
	"id": "randomEncounter",
	"type": "table",
	"label": "随机遭遇",
	"visibility": ["ai", "trigger"]
}
```

## 工具请求

AI 或触发器请求工具时统一成 `ToolRequest`：

```json
{
	"requestId": "tool_001",
	"source": "ai",
	"toolId": "rollD20",
	"reason": "潜行检定",
	"input": {
		"difficulty": 12
	}
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `requestId` | 本轮工具请求 ID，由 GameMode 生成或补齐 |
| `source` | `ai`、`manual`、`trigger` |
| `toolId` | 机制包工具 ID |
| `reason` | 请求原因，用于展示和回传 AI |
| `input` | 工具输入参数 |

## 工具结果

所有工具都返回统一 `ToolResult`：

```json
{
	"requestId": "tool_001",
	"toolId": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"source": "ai",
	"reason": "潜行检定",
	"ok": true,
	"displayText": "D20 检定：1d20 = 7",
	"data": {
		"notation": "1d20",
		"rolls": [7],
		"modifier": 0,
		"total": 7,
		"difficulty": 12,
		"outcome": "failure"
	},
	"patch": {},
	"changes": []
}
```

失败时：

```json
{
	"requestId": "tool_002",
	"toolId": "unknownTool",
	"source": "ai",
	"ok": false,
	"error": {
		"code": "TOOL_NOT_FOUND",
		"message": "工具不存在：unknownTool"
	}
}
```

工具失败不应该让整个回合崩溃。失败结果应回传给 AI，让 AI 改用可用工具或直接给出最终叙述。

## 工具类型

### `dice`

用于骰子检定。

```json
{
	"id": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"config": {
		"notation": "1d20"
	}
}
```

兼容简写：

```json
{
	"id": "rollD20",
	"type": "dice",
	"label": "D20 检定",
	"notation": "1d20"
}
```

支持格式：

```text
1d20
2d6
1d20+2
1d100-10
```

可选输入：

```json
{
	"reason": "潜行检定",
	"difficulty": 12,
	"modifier": 2
}
```

`modifier` 如果存在，可以叠加到固定 `notation` 的修正值上。

### `table`

用于随机表。`table` 不应只理解为“抽一段话”，而应支持权重、条件、标签、子词条和状态影响。

```json
{
	"id": "strangeEvent",
	"type": "table",
	"label": "异常事件",
	"entries": [
		{
			"id": "fogFace",
			"weight": 4,
			"text": "窗外的雾气凝成一张脸。",
			"tags": ["horror", "vision"]
		},
		{
			"id": "lostPage",
			"weight": 1,
			"text": "获得一页日志。",
			"patch": { "investigator.clues": ["日志残页"] }
		}
	]
}
```

`weight` 是权重。`patch` 会写入状态。`condition` 决定词条是否可用：

```json
{
	"weight": 2,
	"text": "巡逻队经过。",
	"condition": { "path": "world.locationTag", "op": "eq", "value": "wild" }
}
```

复杂随机表、子词条、随机池、洗牌袋、保底权重和预算式随机见：[随机池与随机表](./随机池与随机表.md)。

### `encounter`

建议新增，用于组合式随机遭遇。它不是从一张表里抽完整句子，而是从多个随机池抽取结构化槽位。

```json
{
	"id": "randomEncounter",
	"type": "encounter",
	"label": "随机遭遇",
	"visibility": ["ai", "manual", "trigger"],
	"config": {
		"rolls": {
			"category": { "pool": "encounter.category" },
			"actor": { "pool": "encounter.actor", "matchTagsFrom": "category" },
			"intent": { "pool": "encounter.intent" },
			"complication": { "pool": "encounter.complication", "chance": 0.5 },
			"reward": { "pool": "encounter.reward", "chance": 0.35 }
		}
	}
}
```

`encounter` 的 `ToolResult.data.rolls` 要保留每个槽位结果，方便 `narrator` 基于真实结构生成最终叙述。


### `stateCheck`

建议新增，用于通用状态检定，不需要 AI 手写比较逻辑。

```json
{
	"id": "checkHpLow",
	"type": "stateCheck",
	"label": "检查生命危险",
	"visibility": ["ai", "trigger"],
	"config": {
		"path": "player.hp",
		"op": "lte",
		"value": 3
	}
}
```

返回：

```json
{
	"ok": true,
	"data": {
		"passed": true,
		"actual": 2,
		"op": "lte",
		"expected": 3
	}
}
```

### `patchState`

建议新增，用于机制包声明安全的状态修改。

```json
{
	"id": "advanceTime",
	"type": "patchState",
	"label": "推进时间",
	"visibility": ["trigger"],
	"config": {
		"patch": {
			"world.timeOfDay": "night"
		}
	}
}
```

AI 不应该默认拥有任意 `patchState` 权限。若开放给 AI，应限制可写路径。

## Tool Runtime API

建议内部函数形态：

```js
executeGameTool({ pack, state, request, context })
```

输入：

| 参数 | 说明 |
|---|---|
| `pack` | 当前机制包 |
| `state` | 当前游戏状态 |
| `request` | 标准化后的 `ToolRequest` |
| `context` | 当前回合上下文，如用户输入、触发器结果、历史工具结果 |

输出：

```js
{
	result,
	state,
	changes
}
```

其中 `result` 是 `ToolResult`。

## 执行顺序

单轮建议顺序：

```text
1. trigger 工具
2. ai 请求工具
3. final statePatch
```

同一路径后写覆盖前写，同时记录 changes：

```json
{
	"path": "world.locationTag",
	"from": "town",
	"to": "wild",
	"source": "tool:randomEncounter"
}
```

## 安全规则

1. AI 只能请求 `visibility` 包含 `ai` 的工具。
2. 手动按钮只能执行 `visibility` 包含 `manual` 的工具。
3. 触发器只能执行 `visibility` 包含 `trigger` 的工具。
4. 未知工具返回失败 `ToolResult`，不抛出到顶层。
5. 单轮工具调用有上限，例如 `maxToolRounds = 3`、`maxToolCalls = 8`。
6. 工具 `patch` 只能写入机制包状态路径。
7. 工具结果必须结构化保存，不能只拼接文本。

## 保存结构

建议每条 assistant 消息保存：

```js
message.metadata.gameEvent = {
	phase: 'final',
	toolRequests: [],
	toolResults: [],
	stateChanges: [],
	triggerResults: []
}
```

这样普通导出可读，完整存档可恢复和调试。
