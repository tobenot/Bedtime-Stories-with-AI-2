# 内容池方案（V2）

> 解决"分叉对话导致大量消息正文（content / reasoning_content）重复存储"的问题。
>
> 前置依赖：建议在《消息懒加载方案》落地后再实施，但**也可独立实施**（在旧 schema 上）。
>
> 状态：方案设计稿，待落地。

---

## 一、问题陈述

### 1.1 现状

`forkChatAt` 在 `chatMethods.js` 中：

```js
messages: cloneMessagesWithNewIds(messagesToKeep)
```

`cloneMessagesWithNewIds` 内部对每条消息做 `JSON.parse(JSON.stringify(msg))`，**深拷贝整条消息（含 content 全文）**，只是换了个 `id`。

### 1.2 用户实际行为带来的放大

用户大量使用分叉功能：
- 一条 2 万字的消息
- 被分叉进 30 个对话
- = **30 份独立的 2 万字副本** ≈ 1.2 MB（光这一条）

### 1.3 痛点

| 维度 | 影响 |
|---|---|
| **磁盘占用** | IDB 体积线性膨胀 |
| **导出文件体积** | 明文导出体积同样膨胀，不利分享和备份 |
| **导入解析时间** | 30 份相同正文都要 parse |
| **内存常驻** | JS 引擎不会自动去重长字符串（短串才会） |
| **每次保存放大** | 在旧 schema 下，整份 stringify 把重复内容反复序列化 |

### 1.4 与《消息懒加载方案》的差异

- 懒加载方案解决的是 **"全量驻留"** → 内存里只放当前对话
- 内容池方案解决的是 **"重复驻留 / 重复存储"** → 即便同时打开多份引用同内容的消息，也只存一份正文

两者**正交**，叠加效果最佳。

## 二、目标

### 2.1 必须达成

- 同一段长 `content` / `reasoning_content` 在磁盘上**只存一份**
- 同一段长正文在内存中**只存一份字符串引用**
- 分叉对话时不再复制正文，只复制元信息和引用
- 导出 / 导入文件支持**池化格式**（自描述、可还原）
- 提供 **"优化存档"按钮**：手动触发对历史数据的去重，并展示前后对比

### 2.2 不破坏的约束

- **新消息、流式中的消息、刚编辑的消息**继续用内联 `content`，**不**强制立即入池
- **流式写入路径完全不变**（`+= content`）
- **编辑消息路径完全不变**（直接赋值）
- 现有所有读消息文本的代码尽量保持习惯（通过统一 `getMessageText(msg)` 适配）

### 2.3 可接受的代价

- 全代码库读 `msg.content` / `msg.reasoning_content` 的地方（约 8 个文件）改成 `getMessageText(msg)` / `getMessageReasoning(msg)`
- 删除消息 / 删除对话时要维护引用计数

## 三、设计

### 3.1 数据形态

#### 消息字段
```js
// 池化的消息
{
  id, role, createdAt, createdAtMs,
  contentRef: 'c0001',                  // 引用池中条目的短 ID
  reasoningRef: 'c0002',                // 同上
  // 注意：不再有 content / reasoning_content 字段
}

// 未池化的消息（新消息、刚编辑的、流式中的）
{
  id, role, createdAt, createdAtMs,
  content: '...',                       // 原样内联
  reasoning_content: '...',
}

// 两种状态可以共存于同一个对话，互不影响
```

#### 内容池
```js
contentPool = {
  'c0001': { text: '完整文本...', refCount: 30 },
  'c0002': { text: '另一段...', refCount: 1 },
  ...
}
```

### 3.2 短 ID 设计

- 不用 hash 当 ID（hash 太长会吃掉体积收益）
- 内部维护一个 `nextPoolId` 单调递增
- ID 形如 `c0001`、`c0002`、...
- 同时维护一个**反向索引** `textHash → poolId`，用于"加入新内容时查重"
  - hash 用 SHA-256 前 8 字节（足够避免误并）
  - 这个反向索引**只在内存中**，不持久化（启动时重建）

### 3.3 入池阈值

只对**净收益为正**的内容入池：

```
满足以下任一条件才入池：
  - text.length >= 256 且会被引用 ≥ 2 次（去重场景）
  - text.length >= 1024（单次出现也值得，因为短 ID 节省可观）
```

低于阈值的内容继续内联，避免池子里塞一堆"好的"、"嗯"。

### 3.4 池的生命周期管理

采用**引用计数 + 标记清扫双保险**：

#### 引用计数（增量维护）
- 入池：`refCount++`
- 删消息 / 编辑消息（脱离引用）：`refCount--`
- `refCount === 0`：从池中删除

#### 标记清扫（兜底，定期 / 优化按钮触发）
- 全量扫描所有消息的 `contentRef` / `reasoningRef`
- 重新计算所有 poolId 的引用数
- 修正引用计数错误，回收漏掉的孤儿条目

### 3.5 持久化（懒加载方案落地后）

```
bs2-chat-db (v3)
  ├─ chats              (元数据，不变)
  ├─ messages           (消息，新增 contentRef / reasoningRef 字段)
  └─ content_pool       (新)
        keyPath: id
        records: { id, text, refCount }
```

### 3.6 持久化（懒加载未落地的兼容版）

如果选择独立先做内容池（不依赖懒加载方案）：

```
bs2-chat-db (v2)
  └─ kv
        ├─ chat_history    (消息可带 contentRef，正文外置)
        ├─ content_pool    (新增 KV 条目，存全部池条目的 JSON)
        └─ current_chat_id
```

> 推荐还是先做懒加载，再做内容池。两者叠加才是完全体。

### 3.7 内存中的池

```js
// AppCore data
{
  contentPool: {
    'c0001': { text, refCount },
    ...
  },
  poolReverseIndex: {       // 仅运行时，不持久化
    [textHash]: poolId
  },
  nextPoolId: 1,
}
```

## 四、API 设计

### 4.1 新增 `src/utils/contentPool.js`

```js
// 入池：返回 poolId（命中已有的或新分配）
putContent(text)              → string poolId

// 取出：返回字符串
getContent(poolId)            → string

// 引用计数
incRef(poolId)                → void
decRef(poolId)                → void                  // 归零自动删除

// 批量
gcUnreferenced()              → number removed

// 全量重建（标记清扫）
rebuildRefCounts(allChats)    → { fixed: number }

// 池序列化 / 反序列化
exportPool()                  → { entries, nextId }
importPool(snapshot)          → void
```

### 4.2 消息访问适配层

```js
// 统一读消息文本：池化 → 解引用；未池化 → 直接返回
getMessageText(msg)           → string
getMessageReasoning(msg)      → string

// 写消息文本（编辑场景）：转成内联状态
setMessageText(msg, text)     → void                  // 把 contentRef 改回 content
setMessageReasoning(msg, t)   → void
```

### 4.3 优化按钮入口

```js
// 扫描整个 chatHistory，把符合阈值的内联消息池化
optimizeChatArchive()         → Promise<{
  beforeBytes, afterBytes,
  poolEntries, dedupedCount,
  savingsPercent,
}>

// 预览（不写入，只算数）
previewOptimization()         → 同上
```

| `pdfExporter.js` | 117, 120 | 同上 |
| `archive.js` | 12, 109 | `getMessageKey` 等比对函数取文本时解引用 |

### 5.2 必须改造的"写"点

| 文件 | 改造 |
|---|---|
| `providers/*.js` 流式 `+= content` | **不改**，因为流式期间消息保持内联状态 |
| `chatMethods.js` 编辑消息（`:388`） | 走 `setMessageText`：自动减引用并改回内联 |
| `chatMethods.js` `forkChatAt` | 不再深拷贝；新消息直接复用 `contentRef` 并 `incRef` |
| `chatMethods.js` 删消息 / 删对话 | 减引用 |

### 5.3 导出 / 导入

#### 导出
- 导出前调用 `optimizeChatArchive()`（或仅"打包成池化"，不修改原数据）
- 输出文件结构：
```json
{
  "meta": { "version": 2, "format": "pooled", ... },
  "contentPool": {
    "c0001": "完整文本...",
    "c0002": "..."
  },
  "chatHistory": [
    {
      "messages": [
        { "role": "user", "contentRef": "c0001" },
        { "role": "assistant", "content": "短消息直接内联" }
      ]
    }
  ]
}
```
- 旧版本兼容：导出时也提供"导出兼容版（明文展开）"选项

#### 导入
- 读 `meta.format`：
  - `pooled`：先读 `contentPool` → 写入应用池 → 消息直接带 `contentRef` 入库
  - `legacy`（无 contentPool 字段）：按现有路径读内联消息
- 导入后**不强制全量优化**（保持"新消息不立即优化"语义）

### 5.4 加密存档

- `secureArchive.js` 在 gzip 之前的 payload 与导出格式一致 → 自动支持池化
- gzip 对重复字符串本身有压缩 → 池化收益在加密导出中部分被吸收，但**导入侧 parse 体积仍受益**

## 六、优化按钮（核心 UX）

### 6.1 入口位置

挂在设置面板（`SettingsDrawer.vue`）的"存档管理"区，与现有"统一修复"按钮并列：

- 按钮文案：**"优化本地存档"**
- 副标题：**"扫描重复内容并去重，节省存储空间"**

### 6.2 交互流程

```
1. 用户点击"优化本地存档"
2. 弹出"扫描中..."进度提示
3. 后台执行 previewOptimization()
4. 弹出确认对话框：
   ┌──────────────────────────────────────┐
   │ 扫描完成                              │
   │                                      │
   │ 当前大小：    18.4 MB                 │
   │ 优化后大小：   5.7 MB                 │
   │ 节省：        12.7 MB (69%)          │
   │                                      │
   │ 去重消息：    142 条                 │
   │ 内容池条目：   23 条                 │
   │                                      │
   │ [取消]              [确认优化]        │
   └──────────────────────────────────────┘
5. 用户确认 → 执行 optimizeChatArchive() 落盘
6. 成功提示 + 自动刷新
```

### 6.3 大小计算口径

- 用 `new Blob([JSON.stringify(payload)]).size`，而不是 `string.length`
- 单位：自动选择 KB / MB
- 节省百分比：`(before - after) / before * 100`

### 6.4 进度反馈

- 总消息数 > 1000 时显示进度条
- 用 `requestIdleCallback` 分片处理，避免阻塞主线程
- 失败时回滚（不写入），并提示用户

## 七、风险与对策

| 风险 | 对策 |
|---|---|
| 流式 `+= content` 误把 `contentRef` 当字符串拼接 | 流式期间消息**保持内联状态**，本来就有 `content` 字段；池化只发生在历史消息上 |
| 编辑消息后池条目变孤儿 | `setMessageText` 内置 decRef + 转内联；下次"优化"按钮可再次入池 |
| 引用计数错乱（漏减、漏加） | `optimizeChatArchive` 内含 `rebuildRefCounts` 标记清扫兜底 |
| 短 ID 在导出文件里偶然冲突（多文件合并） | 导入时**重新分配 ID**，建立 `oldId → newId` 映射 |
| 加载时 contentRef 找不到对应池条目 | 降级为空字符串 + 控制台 warning + 不影响其他消息 |
| 池条目巨大（单条 5 万字） | 不影响功能，但建议在统计面板告警提醒 |
| 用户在多端同步存档（未来） | 池条目按 hash 命名可实现幂等合并；当前 V2 不考虑 |
| 与导出兼容性 | 导出菜单提供"池化版（推荐，体积小）"和"明文版（兼容旧版）"两个选项 |

## 八、性能预期

### 8.1 体积（300 对话场景，假设 30% 内容来自分叉冗余）

| 指标 | 现状 | 池化后 |
|---|---|---|
| IDB 占用 | ~30 MB | ~21 MB（去重 30%） |
| 明文导出文件 | ~30 MB | ~21 MB |
| 加密导出文件（gzip 后） | ~6 MB | ~5 MB（gzip 已吸收部分重复） |
| 导入 parse 时间 | ~300ms | ~210ms |

### 8.2 内存（独立做内容池时）

| 指标 | 现状 | 池化后 |
|---|---|---|
| 长 content 内存占用 | 每份独立 | 同 hash 共享 1 份 |
| 估算节省 | - | 约 30% 长文本内存 |

> 内存收益的真正大头还是来自《消息懒加载方案》，本方案的内存收益是锦上添花。

## 十、与《消息懒加载方案》的协作关系

### 推荐顺序：先懒加载，后内容池

理由：
1. 懒加载落地后，`messages` store 已经分行 → 加 `contentRef` 字段成本极低
2. 懒加载会让"加载某对话时按需解引用"成为天然路径，运行时压力小
3. 懒加载已经把内存压力解决掉了 → 内容池可以专注解决"磁盘 / 导出体积"

### 独立实施的可能性

如果出于某种原因要先做内容池（比如要先解决导出体积问题），也可以在当前 schema 上独立做，只是：
- 没有"按对话粒度加载"，全量加载时所有池条目都驻留内存
- 改动面相对集中，但缺乏懒加载的辅助优化
