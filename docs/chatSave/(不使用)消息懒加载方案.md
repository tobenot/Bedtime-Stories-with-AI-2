# 消息懒加载方案（V1）

> 解决"几百个对话全量驻留内存"导致的内存膨胀、保存耗时膨胀、启动变慢三大问题。
>
> 状态：方案设计稿，待落地。

---

## 一、问题陈述

### 1.1 现状

应用启动时，`AppCore.vue` 把 IndexedDB 中**全部对话**一次性 `JSON.parse` 到 `this.chatHistory`：

```js
// src/AppCore.vue
data() {
  return {
    chatHistory: [],   // 整个 N 对话 × M 消息的数组
    currentChatId: null,
  }
}
```

`chatStorage.js` 的实现是单 KV 表：

```
bs2-chat-db / kv
  ├─ chat_history     → 整份 JSON 字符串
  └─ current_chat_id  → 当前对话 ID
```

`chatMethods.saveChatHistory()` 每次保存：

```js
this.chatHistory = sortChatsByCreatedTime(this.chatHistory);
await this.tryPersistChatHistory(this.chatHistory);  // JSON.stringify 整份 → IDB.put 整份
```

### 1.2 实际量级（以 300 对话为例）

| 项目 | 估算 |
|---|---|
| 对话数 | 300 |
| 平均消息数 / 对话 | 20 |
| 总消息数 | 6000 |
| 平均消息长度 | 1000 字 |
| 总字数 | 600 万 |
| 纯文本字节（UTF-8 中文 3 字节） | ≈ 18 MB |
| 加 `reasoning_content` | ≈ 30 MB |
| Vue 响应式（Proxy）开销 | × 2~3 倍 |
| **JS 堆常驻** | **≈ 60–100 MB** |

### 1.3 三大痛点

1. **内存膨胀**：用户根本没打开的 295 个对话的全部消息都在 RAM 里、且都被 Vue 响应式追踪
2. **保存放大**：用户每发一句话，整份 30 MB 都要重新 `stringify` + 写 IDB（O(总历史) 而非 O(变更量)）
3. **启动慢**：冷启动一次性 `JSON.parse(30MB)` 直接顶在白屏到可交互之间

侧边栏 `ChatSidebar.vue` 实际只需要 `{ id, title, mode, createdAtMs }`，**完全不需要 messages**——但目前所有 messages 都被一起拉进来。

## 二、目标

### 2.1 必须达成

- 内存中**只驻留对话元数据 + 当前对话的 messages**
- 切换对话时按需加载该对话的 messages
- 保存某条对话时**只写该对话**，不重写其他对话
- 启动时间从 O(总历史) 降到 O(对话数)

### 2.2 不破坏的约束

- **导出 / 导入存档格式不变**（仍是当前的内联结构，与"内容池"方案解耦）
- **流式写入路径不变**（`provider` 直接 `+= content`）
- **编辑消息路径不变**（直接赋值 `msg.content`）
- **加密存档不变**
- 数据修复（`normalizeAndRepairChats`）仍然可用

### 2.3 可接受的代价

- 切换对话时若未命中缓存，会有一次异步加载（目标 < 50ms）
- 部分调用方需要从同步取消息改为 `await ensureMessagesLoaded(chatId)`

## 三、设计

### 3.1 IndexedDB Schema 升级

`CHAT_DB_VERSION` 从 `1` → `2`，在 `onupgradeneeded` 里新增两个 store：

```
bs2-chat-db (v2)
  ├─ kv                 (旧 store，兼容用 + 存 currentChatId)
  │     └─ current_chat_id
  │
  ├─ chats              (新 store，对话元数据)
  │     keyPath: id
  │     records: {
  │       id, title, isTitleManuallyEdited,
  │       mode, createdAt, createdAtMs,
  │       messageCount,           // 冗余字段，方便侧边栏显示
  │       lastMessageAt,          // 排序用
  │       protection,             // 密码保护信息
  │       hasMessages: true,      // 区分"已迁移"与"空对话"
  │     }
  │
  └─ messages           (新 store，按对话存消息)
        keyPath: id
        index 'by_chat': chatId
        records: {
          id, chatId,
          role, content, reasoning_content,
          createdAt, createdAtMs,
          orderIndex,             // 同一对话内的顺序（避免依赖时间戳排序）
          // 其余 UI 状态字段（isReasoningCollapsed 等）
        }
```

> `chat_history` 这个 key 在迁移完成后会被删除。

### 3.2 内存模型

```js
// AppCore data
{
  chatMetaList: [          // 全部对话的元数据（替代原 chatHistory）
    { id, title, mode, createdAtMs, messageCount, ... }
  ],

  loadedMessages: {        // 已加载的对话消息（LRU）
    [chatId]: Array<Message>
  },

  currentChatId: string,
}
```

### 3.3 核心 computed 改造

```js
// 之前
currentChat() {
  return this.chatHistory.find(c => c.id === this.currentChatId);
}

// 之后
currentChat() {
  const meta = this.chatMetaList.find(c => c.id === this.currentChatId);
  if (!meta) return null;
  return {
    ...meta,
    messages: this.loadedMessages[this.currentChatId] || [],
    _loaded: !!this.loadedMessages[this.currentChatId],
  };
}
```

> `_loaded` 给 UI 用：未加载完时可以显示骨架屏。

### 3.4 LRU 缓存策略

- 默认保留**最近打开的 5 个对话**的 messages
- 切换对话时：
  1. 把目标对话标记为 MRU
  2. 异步加载（若未命中）
  3. 触发 LRU 淘汰（超过 5 个时移除最旧的）
- 不淘汰：当前对话、有 unsaved 改动的对话、正在流式输出的对话

### 3.5 写入策略

不再有"全量保存"。所有写入都按粒度拆分：

| 触发场景 | 写入操作 |
|---|---|
| 用户发消息 | `messages.put(newMessage)` + 更新所属 chat 的 `messageCount` / `lastMessageAt` |
| 流式 token 追加 | **不立即写**，节流 500ms 写一次 `messages.put(streamingMessage)` |
| 流式结束 | `messages.put(finalMessage)` + 更新 chat meta |
| 编辑消息 | `messages.put(editedMessage)` |
| 新建对话 | `chats.put(newChatMeta)` |
| 删除对话 | `chats.delete(id)` + `messages.deleteByIndex('by_chat', id)` |
| 重命名 | `chats.put(updatedMeta)` |
| 切换 currentChatId | `kv.put('current_chat_id', id)`（保持现有逻辑） |

所有写入都走现有 `enqueueChatStorageTask` 队列，保持串行化。

### 4.1 新增 `src/utils/chatStorage.js` 接口

```js

// currentChatId（保持原样）
loadCurrentChatId()         → Promise<string|null>

// 整体

// 迁移

```

### 4.2 新增 `chatMethods` 上的方法

```js
async ensureMessagesLoaded(chatId)         // 幂等：已加载直接返回
async switchChat(chatId)                   // 切换 + 加载 + LRU 维护
async preloadAdjacentChats()               // 可选：侧边栏 hover 触发
evictLruMessages()                         // 内部 LRU 淘汰
```

### 4.3 改造的方法

| 方法 | 改造点 |
|---|---|
| `loadChatHistory()` | 改为加载 meta 列表 + 当前 chat 的 messages |
| `saveChatHistory()` | **废弃**，调用方改用细粒度方法 |
| `createChatRecord()` | 写 meta + 当前消息（如果有） |

## 五、迁移方案

### 5.1 触发时机

`onupgradeneeded` 里建好新 store 后，**首次** `loadChatHistory` 时：
1. 检测 `chats` store 是否为空 + `kv.chat_history` 是否存在
2. 若是 → 走迁移流程
3. 迁移完成后**删除** `kv.chat_history`，避免下次再迁

### 5.2 迁移流程

```
1. 从 kv 读出旧的 chat_history JSON 字符串
2. JSON.parse → 走一遍 normalizeAndRepairChats（沿用现有修复逻辑）
3. 在一个事务里：
   - 对每个 chat：
     - 拆出元数据写入 chats store
     - 把 messages 数组每条写入 messages store（带 orderIndex）
4. 事务提交后，从 kv 删除 chat_history key
5. 在控制台输出迁移结果
```

### 5.3 失败回滚

- 迁移用单事务，失败自动回滚
- 失败时**保留** `kv.chat_history`，下次启动重试
- 连续失败 3 次 → 弹窗提示用户导出存档后联系作者

### 5.4 兼容旧版本回退

如果用户从新版回退到旧版（用了 v2 后又装回 v1）：
- 旧版只读 `kv.chat_history` → 读不到 → 显示空
- **保护策略**：迁移成功后，**保留 `kv.chat_history` 一周**作为应急回退副本，一周后才真正删除（用 `kv.migration_legacy_kept_until` 时间戳记录）

## 十、与"内容池方案"的关系

- **本方案是 V1**，先解决"全量驻留 + 全量保存"两个根本性问题
- **内容池是 V2**，在本方案落地后，给 `messages` store 增加 `contentRef` 字段，把分叉冗余的 `content` 池化
- 两个方案**完全解耦**，分开发布、独立可回滚
- V2 实施成本因为 V1 把消息分行了，会变得比直接做更简单

## 附录：工程坑梳理（讨论记录）

> 以下是对本方案进行逐项讨论后发现的工程难点和潜在风险，记录于此供后续落地参考。

### A. messages store 的 keyPath 与 chatId 问题

方案要求每条 message 有 `chatId` 字段（用于 `by_chat` 索引），但现有代码中创建 message 的地方**都没有 chatId**：

- `chatMethods.createMessage()` — 只有 `id, role, content, createdAt, createdAtMs`
- `StandardChatMode.vue` — 手动构造 `userMessage` / `assistantMessage`，无 chatId
- `DrawMode.vue` — 同上
- `forkChatAt` 中的 `cloneMessagesWithNewIds` — 深拷贝但不注入 chatId

**影响**：
1. 迁移时需遍历所有 chat，给每条 message 注入所属 chat.id 作为 chatId
2. 改造后所有创建消息的地方（包括模式组件中手动构造的）都要带上 chatId
3. `cloneMessagesWithNewIds` 克隆时要修正 chatId 为新对话的 id

### B. `mergeImportedChats` 改造复杂度被严重低估

方案第六节表格中写道 `archive.js 导入：拆成 meta + messages 分别写入`，实际远比一句话复杂：

#### B.1 现有实现的精细之处

`mergeImportedChats` 做了以下细致工作：
1. 按 ID 匹配 — `existingById` Map, O(1) 查找
2. 标题合并策略 — `pickMergedTitle` 判断手动标题 vs 自动标题
3. 消息取长 — 同 ID 对话，`importedMessages.length > targetMessages.length` 时取导入版
4. 时间戳取早 — `createdAt` / `createdAtMs` 取更早的
5. 新对话插头部 — `unshift` + `generateUniqueBranchTitle` 去重
6. 倒序遍历保序 — `[...importedChats].reverse()` 保证 unshift 后顺序正确
7. 就地修改 — 直接操作 existingChats 数组

#### B.2 改造后面临的问题

**消息取长需要已有对话的 messages**：改造后已有对话的 messages 在 IDB 的 `messages` store 里，不在内存中。要么全加载（回到内存高峰），要么只比 count（用 IDB index 的 count()），要么放弃取长策略。

**标题合并依赖消息内容**：`pickMergedTitle` → `isManualTitle` → `getDefaultAutoTitle` 需要读 `chat.messages` 中第一条 user 消息的前 20 字符来判断标题是否为自动生成的。改造后也需要从 IDB 读。

**不同 ID 的新对话（设备 A/B 互导场景）写回问题**：`mergeImportedChats` 就地 `unshift` 到 existingChats 数组中。写回 IDB 时需要区分哪些 chat 是"新增的"（需要写入 meta + messages）、哪些是"被更新的"（需要更新 meta，可能需要替换 messages）、哪些是"没动过的"（不需要碰）。如果无脑全量写回，没加载 messages 的 chat 会被写成空 messages。

#### B.3 可行的改造方向

有两种路线：

**路线 A — 临时膨胀**：导入时把命中的 existing chat 的 messages 从 IDB 加载出来，拼回完整的内联结构，用现有 `mergeImportedChats` 逻辑处理，再拆分写回。核心逻辑不改，但需要精确区分"脏"对象只写变化部分。

**路线 B — 生成 patch list**：把 `mergeImportedChats` 从"就地修改大数组"改为"生成变更操作列表（insert/update）"，调用层根据 action list 精确读写 IDB。核心合并逻辑（pickMergedTitle、消息取长、generateUniqueBranchTitle）保留，但输出格式变了。

### C. `saveChatHistory` 有 19 处调用

方案说"废弃 `saveChatHistory`，调用方改用细粒度写入"，实际 `saveChatHistory()` 在以下位置被调用：

| 文件 | 调用数 | 场景 |
|---|---|---|
| `AppCore.vue` | 3 | `@update-chat` 事件、`pagehide`、`visibilitychange` |
| `chatMethods.js` | 11 | 创建/删除/切换对话、修改标题、分叉、编辑消息、删除消息、折叠推理、折叠消息、重新生成、数据修复 |
| `archiveMethods.js` | 2 | 修复数据、导入后 |
| `configMethods.js` | 1 | 切换模式 |

每一处都需要分析"它实际修改了什么"，然后替换为对应的细粒度写入。例如 `toggleReasoning` 只改了一条消息的 `isReasoningCollapsed`，应该替换为 `saveMessage(msg)`；而 `deleteChat` 需要 `deleteChatMeta + deleteMessagesByChat`。

这是一个逐个击破的体力活，但错一个就会丢数据或产生不一致。

### D. 内存模型切换的连锁反应

方案要求 `chatHistory` → `chatMetaList`，`currentChat` 改为 computed 拼接。但：

- `ChatSidebar` 接收的 prop 是 `chatHistory`，需要改为 `chatMetaList`（不含 messages，但现有侧边栏确实不用 messages，影响小）
- `currentChat` 变为异步依赖（messages 可能未加载），所有**同步读取** `currentChat.messages` 的地方都要加 `_loaded` 判断或 `await ensureMessagesLoaded`
- 流式输出期间 `provider` 直接 `+= content` 到 `currentChat.messages[last].content`，如果 currentChat 是 computed 拼接的，需要确保 `loadedMessages[chatId]` 的引用稳定性

### E. 迁移的原子性与回退

- 方案设计了单事务迁移 + 保留旧数据 1 周，这部分设计合理
- 但如果迁移成功后用户在新 schema 上使用了一段时间，再回退旧版，旧版只能看到 1 周前的快照，中间的变更全丢
- 这不是 bug，但需要在 UI 上明确告知用户"升级后请勿回退"

### F. 总结：为什么不建议一步到位

综合以上坑点，完整的懒加载方案改造量大（特别是 B 和 C），风险集中在数据一致性上。建议分两期：

- **第一期**：只把 messages 从全量保存中拆出来存 IDB 独立 store，解决写入瓶颈（O(全量) → O(单对话)）。启动时仍全量加载到内存，**不做懒加载**，所有现有逻辑（包括 mergeImportedChats）不动。
- **第二期**：等数据量真正逼近内存/启动瓶颈时，再做懒加载 + LRU + 异步切换。

详见独立方案文档：`docs/Messages摘出方案.md`。
