# 冷热分离方案（对话归档）

> 通过用户主动归档，将不常用的旧对话从主存储区移入独立的归档区，实现**启动提速、内存瘦身、保存加速**三重收益，同时**对现有代码几乎零改动**。
>
> 状态：方案设计稿，待讨论确认。

---

## 一、问题陈述

### 1.1 现状

应用启动时，`chatStorage.js` 从 IndexedDB 的单条 KV 中读出**全部对话**的 JSON 字符串，`chatMethods.loadChatHistory()` 一次性 `JSON.parse` 到 `this.chatHistory`：

```
bs2-chat-db (v1) / kv
  ├─ chat_history     → 整份 JSON 字符串（含所有对话的所有消息）
  └─ current_chat_id  → 当前对话 ID
```

`saveChatHistory()` 每次保存时 `JSON.stringify` 整份 `chatHistory` + IDB.put 整份。

### 1.2 痛点

| 痛点 | 说明 |
|---|---|
| **启动慢** | 冷启动一次性 `JSON.parse(30MB)` 顶在白屏到可交互之间 |
| **内存膨胀** | 用户没打开的 295 个对话的全部消息都在 RAM 里、且都被 Vue 响应式追踪 |
| **保存放大** | 用户每发一句话，整份 30MB 都要重新 `stringify` + 写 IDB |

### 1.3 核心思路

用户主动（或自动）将旧对话**归档**到独立存储区。归档后：

- 热区（侧边栏里看得到的对话）体积大幅缩小 → 启动快、保存快、内存小
- 冷区（用户自己收起来的对话）不参与日常加载和保存
- 用户随时可以取回归档对话

这是经典的**冷热分离**策略，也是用户最容易理解的交互模式——跟 Gmail 的归档、微信的折叠群聊、手机相册的归档照片一模一样。

## 二、目标

### 2.1 必须达成

- 启动时只加载热区数据，启动时间从 O(总历史) 降到 O(热数据)
- 内存中只驻留热区的对话（含 messages）
- 保存时只序列化热区数据
- 归档/取回操作用户可见可控
- 现有代码改动量最小化

### 2.2 不改变的东西

- **`this.chatHistory` 数据结构不变** — 仍是完整的 chat 数组（含 messages），只是内容少了
- **`currentChat` computed 不变** — `this.chatHistory.find(...)`
- **`saveChatHistory()` 不变** — 仍是全量 stringify 热区，但数据量小了
- **`mergeImportedChats` 不变** — 只跟热区比对
- **`normalizeAndRepairChats` 不变** — 只修复热区
- **流式写入路径不变** — `+= content`
- **编辑消息路径不变** — 直接赋值 `msg.content`
- **导入/导出存档格式**：完整备份导出格式升级为 v2（冷热分离），向前兼容旧格式导入
- **加密存档不变**

### 2.3 可接受的代价

- IDB schema 升级到 v2，新增一个 `chatArchive` store
- 归档/取回操作涉及异步 IDB 读写（但都是用户主动触发的低频操作）
- 侧边栏需要增加归档区入口

## 三、用户心智模型

### 3.1 核心原则

> **用户手动归档 = 用户自己画了一条线。**
> **这条线两边的数据，在所有操作中都应该默认分开。**
> **只有用户明确说"我要全部"的时候，才合并。**

在用户眼中，世界是这样的：

```
我的对话（侧边栏里看得到的）  ← 热区，"正在用的"
归档的对话（我自己收起来的）  ← 冷区，"收起来了但没删"
```

### 3.2 所有操作遵循此心智

| 操作 | 用户理解 | 实现 |
|---|---|---|
| 导出全部对话 | "把我侧边栏里的都导出" | 只导热区 |
| 导出最近 80 条 | "把我最近聊的导出" | 只导热区 |
| 导出当前对话 | "只要这一条" | 不变 |
| 导出归档 | "把我收起来的也拿出来" | 只导冷区 |
| 导出完整备份 | "我要全量备份" | 热+冷分开导出（保留冷热结构） |
| 导入存档 | "导入的东西应该出现在侧边栏里" | 只跟热区比对，结果写入热区 |
| 搜索对话 | "搜我正在用的" | 默认只搜热区 |
| 数据修复 | "修修我正在用的" | 只修热区 |
| 标题搜索 | "在侧边栏里找" | 只搜热区 |

## 四、设计

### 4.1 IndexedDB Schema 升级

`CHAT_DB_VERSION` 从 `1` → `2`，在 `onupgradeneeded` 里新增 store：

```
bs2-chat-db (v2)
  ├─ kv                  (旧 store，保留)
  │     ├─ chat_history        → 热区对话的 JSON 字符串
  │     ├─ current_chat_id     → 当前对话 ID
  │     └─ archive_index       → 冷区轻量索引（新增 key）
  │
  └─ chatArchive         (新 store，归档对话)
        keyPath: 无（out-of-line key，用 chat.id 作 key）
        records: 完整的 chat 对象（含 messages），与 chatHistory 中的结构一致
```

**设计要点：**

- 热区仍然用现有的 `kv.chat_history` 存储，**完全不改现有存储逻辑**
- 冷区新增一个 `chatArchive` store，每条对话独立存储（key = chat.id）
- 额外维护一个轻量级 `archive_index`，存放冷区对话的标题索引，用于侧边栏展示

### 4.2 归档索引结构

```js
// kv.archive_index 的值
[
  { id: 'uuid-1', title: '3月份的对话', createdAtMs: 1711234567890 },
  { id: 'uuid-2', title: '2月份的对话', createdAtMs: 1709234567890 },
  // ...
]
```

每条索引项只有 ~100 字节，1000 条归档对话的索引也才 ~100 KB，可以放心全量加载。

### 4.3 内存模型（不变！）

```js
// AppCore data — 完全不改
{
  chatHistory: [],       // 仍然是完整的 chat 数组（含 messages），只是只有热数据
  currentChatId: null,
  archiveIndex: [],      // 新增：冷区标题索引（轻量，启动时加载）
}

// currentChat computed — 完全不改
currentChat() {
  return this.chatHistory.find(chat => chat.id === this.currentChatId);
}
```

### 4.4 核心操作

只需三个操作：

#### 操作 1：归档（热→冷）

```js
async archiveChat(chatId) {
  const index = this.chatHistory.findIndex(c => c.id === chatId);
  if (index === -1) return;
  
  const chat = this.chatHistory[index];
  
  // 1. 写入冷区 store
  await idb.put('chatArchive', chat, chat.id);
  
  // 2. 更新归档索引
  this.archiveIndex.push({
    id: chat.id,
    title: chat.title || '新对话',
    createdAtMs: chat.createdAtMs || Date.now()
  });
  await this.saveArchiveIndex();
  
  // 3. 从热区移除
  this.chatHistory.splice(index, 1);
  
  // 4. 如果归档的是当前对话，切换到第一条热对话
  if (this.currentChatId === chatId) {
    if (this.chatHistory.length > 0) {
      this.currentChatId = this.chatHistory[0].id;
    } else {
      this.createNewChat();
    }
  }
  
  // 5. 保存热区（现有逻辑，数据量变小了）
  this.saveChatHistory();
}
```

#### 操作 2：取回（冷→热）

```js
async restoreChat(chatId) {
  // 1. 从冷区读取完整对话
  const chat = await idb.get('chatArchive', chatId);
  if (!chat) return;
  
  // 2. 放回热区
  this.chatHistory.push(chat);
  this.chatHistory = sortChatsByCreatedTime(this.chatHistory);
  
  // 3. 从冷区删除
  await idb.delete('chatArchive', chatId);
  
  // 4. 更新归档索引
  this.archiveIndex = this.archiveIndex.filter(item => item.id !== chatId);
  await this.saveArchiveIndex();
  
  // 5. 切换到取回的对话
  this.currentChatId = chat.id;
  this.persistCurrentChatId(chat.id);
  
  // 6. 保存热区
  this.saveChatHistory();
}
```

#### 操作 3：批量归档

```js
async archiveOldChats(threshold = 50) {
  if (this.chatHistory.length <= threshold) return;
  
  const sorted = sortChatsByCreatedTime(this.chatHistory);
  const hot = sorted.slice(0, threshold);      // 最近的留着
  const cold = sorted.slice(threshold);          // 老的归档
  
  // 批量写入冷区
  for (const chat of cold) {
    await idb.put('chatArchive', chat, chat.id);
    this.archiveIndex.push({
      id: chat.id,
      title: chat.title || '新对话',
      createdAtMs: chat.createdAtMs || Date.now()
    });
  }
  await this.saveArchiveIndex();
  
  // 热区只保留最近的
  this.chatHistory = hot;
  
  // 确保当前对话在热区
  if (!hot.find(c => c.id === this.currentChatId)) {
    this.currentChatId = hot[0]?.id || null;
    if (!this.currentChatId) this.createNewChat();
  }
  
  this.saveChatHistory();
}
```

### 6.3 导入

#### 普通存档导入（行为不变）

```
用户导入一份文件
  → 解析 → normalizeAndRepairChats (不改)
  → 按 importMode:
     覆盖: 赋值给 chatHistory (不改)
     合并: mergeImportedChats(imported, this.chatHistory) (不改)
  → saveChatHistory() (不改)
```

**导入只跟热区比对**。理由：

- 用户的心智是"导入的东西应该出现在侧边栏里"
- 如果导入的对话 ID 跟冷区某条重复——说明用户之前归档了它，现在又导入了一份（可能从另一台设备来的）。合理的做法是当作新数据放到热区，冷区那份不动
- 这样 `mergeImportedChats` **完全不需要改**

#### 完整备份导入（新增 — 冷热分区恢复）

```
用户导入 full_backup v2 格式文件
  → parseArchiveJson 识别 isFullBackup
  → handleFullBackupImport():
     覆盖模式:
       1. normalizeAndRepairChats(hotChats) → sortedHot
       2. normalizeAndRepairChats(archivedChats) → normalizedCold
       3. 从 normalizedCold 重建 archiveIndex
       4. replaceAllStorageData() 原子替换热区+冷区+索引+currentChatId
       5. 更新内存状态
     合并模式:
       1. mergeImportedChats(hotChats, this.chatHistory)
       2. getAllArchivedChats() → 按 ID 去重合并冷区
       3. 重建 archiveIndex
       4. replaceAllStorageData() 原子写入
       5. 更新内存状态
```

**向前兼容**：旧版 `full_backup`（v1，冷热合并到 chatHistory）仍走普通导入路径。

#### 完整备份导入核心 API

```js
// chatStorage.js
replaceAllStorageData({ hotHistory, archivedChats, archiveIndex, currentChatId })
  → 一个 IDB 事务：清空冷区 → 逐条写入冷区 → 覆盖热区 → 写入索引 → 写入 currentChatId
```

## 九、迁移方案

### 9.1 触发时机

`onupgradeneeded`（version 1 → 2）里建好 `chatArchive` store。

### 9.2 不需要数据迁移！

**这是本方案最大的优势。**

冷热分离不需要迁移任何现有数据：

- 现有的 `kv.chat_history` 就是热区（升级时默认所有对话都是热数据）
- `chatArchive` store 初始为空
- 用户在使用过程中逐步归档旧对话，冷区自然填充

对比 Messages 摘出方案需要拆分所有 chat 到两个 store + 注入 chatId + orderIndex，本方案**零迁移风险**。

### 9.3 版本回退兼容

- 如果用户从 v2 回退到 v1：
  - v1 只读 `kv.chat_history` → 正常读到热区数据
  - 冷区数据在 `chatArchive` store 里，v1 不读它但也不会删它
  - 用户看到的是"没有被归档的那些对话"——**不丢数据，只是归档的看不到**
  - 再次升级到 v2 时，冷区数据完好无损

## 十、性能预期

以 300 条对话、热区阈值 50 条为例：

| 指标 | 现状 | 冷热分离后 |
|---|---|---|
| 启动 parse 数据量 | ~30 MB（300 条全量） | ~5 MB（50 条热数据） |
| 启动 parse 时间 | ~300ms | ~50ms |
| 单次 saveChatHistory 耗时 | ~200ms（stringify 30MB） | ~33ms（stringify 5MB） |
| 内存常驻 | 60–100 MB | 10–17 MB |
| 归档操作耗时 | — | ~10ms（单条 IDB put） |
| 取回操作耗时 | — | ~10ms（单条 IDB get） |
| 批量归档耗时 | — | ~500ms（250 条） |

## 十一、与其他方案的关系

### 11.1 对比

| 维度 | Messages 摘出方案 | 消息懒加载方案 | **冷热分离方案** |
|---|---|---|---|
| 改造范围 | 19 处 saveChatHistory 替换 | 全面重构内存模型 | **新增功能，现有代码不动** |
| 内存模型 | 不变 | 彻底重构 | **不变** |
| mergeImportedChats | 不改 | 需要大改 | **不改** |
| 解决启动慢 | ❌ | ✅ | **✅** |
| 解决内存占用 | ❌ | ✅ | **✅** |
| 解决保存慢 | ✅（根本性） | ✅ | **✅（间接，因为数据量小了）** |
| 迁移复杂度 | 中（拆分 + 注入 chatId） | 高 | **零** |
| 风险 | 中（写入链路变了） | 高（异步化连锁反应） | **低** |
| 用户感知 | 无 | 无 | **有**（归档是用户操作） |

### 11.2 独立还是组合？

本方案**完全独立于**其他方案，可以单独实施。

但也可以作为渐进式优化的**第一步**：

```
第一步：冷热分离（本方案）
  → 以极低成本解决启动慢、内存大、保存慢三大问题
  → 给用户一个"归档"功能

第二步：Messages 摘出方案（可选）
  → 如果热区本身也很大（比如单条对话有上千条消息），再优化保存链路

第三步：消息懒加载（可选）
  → 如果单条对话的消息极多，再做按需加载

第四步：内容池（可选）
  → 如果分叉导致大量重复内容，再做去重
```

**每一步都独立可回滚，不形成耦合。**

## 十二、风险与对策

| 风险 | 对策 |
|---|---|
| 用户误归档了正在用的对话 | 归档时确认弹窗；归档后 toast 提示"已归档，可在底部取回" |
| 归档索引与 chatArchive store 不一致 | 打开归档管理时校验一次；提供"重建索引"按钮 |
| 冷区对话数据损坏 | 每条对话独立存储，单条损坏不影响其他；取回时校验 |
| IDB 存储空间限制 | 热区+冷区总量不变，不会因分离增加空间占用 |
| 用户回退旧版看不到归档 | v1 只读 `kv.chat_history`（热区），归档数据保持不动不丢失 |
| 自动归档误将用户活跃的对话归档 | 自动归档只按时间排序归档最老的；P0 阶段只做手动归档 |
| 批量归档时当前对话被归档 | 代码中跳过 `currentChatId` 对应的对话 |
| 导入大量数据后热区膨胀 | 导入后提示用户"您有 N 条对话，是否归档旧的？" |
